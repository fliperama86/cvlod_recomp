## Project Overview

This is a **static binary recompilation project** for Castlevania: Legacy of Darkness (N64). The goal is to convert the original N64 MIPS binary into native C code using N64Recomp, enabling the game to run on modern systems.

**Current Status:** 100% recompilation achieved - 16,646 functions converted to C without logic stubs.

## Build Commands

```bash
# Full self-healing build pipeline (recommended)
python3 tools/super_build.py

# Individual steps (if needed):
python3 -m splat split config/castlevania2.yaml   # Disassemble ROM (creates asm/)
python3 tools/scan_symbols.py                      # Find missing cross-file symbols
python3 tools/scan_stubs.py                        # Find unsupported instructions
python3 tools/patch_asm.py                         # Apply stubs and fix relocations
python3 tools/build.py                             # Parallel assembly and linking

# Run N64Recomp manually
lib/N64Recomp/build/N64Recomp recomp.toml
```

The build uses Docker with `debian:bullseye-slim` and the `mips-linux-gnu` toolchain. The `build.py` script uses multiprocessing for parallel assembly.

## Architecture

### Build Pipeline

```
ROM (castlevania_legacy_of_darkness.z64)
    ↓ splat
MIPS Assembly (asm/)
    ↓ scan_symbols.py + scan_stubs.py + patch_asm.py
Patched Assembly
    ↓ build.py (mips-linux-gnu-as/ld)
ELF Binary (build/castlevania2.elf)
    ↓ N64Recomp
Native C Code (recompiled/)
```

### Key Configuration Files

- **recomp.toml** - N64Recomp configuration; critical setting: `use_lookup_for_all_function_calls = true` enables dynamic function resolution
- **config/castlevania2.yaml** - Splat config defining ROM structure and segment mapping
- **config/castlevania2.ld** - Linker script (~3500 lines) mapping memory layout
- **symbols/symbol_addrs.txt** - Global symbol map for the binary
- **symbols/stubs.txt** - Functions to stub (hardware ops, unsupported patterns)

### Self-Healing Build System

The build system automatically handles errors:
1. **tools/scan_symbols.py** - Pre-emptively finds cross-file labels (`.L...` symbols)
2. **tools/scan_stubs.py** - Identifies hardware instructions unsupported by N64Recomp (COP0, cache, TLB, eret, etc.)
3. **tools/patch_asm.py** - Applies stubs and fixes relocation truncation errors
4. **tools/super_build.py** - Orchestrates everything, auto-stubs failing functions on N64Recomp errors

### Directory Structure

- `asm/` - Disassembled MIPS code (generated by splat)
- `assets/` - Game resources and binary assets (ipl3.bin in assets/bin/)
- `build/` - Compiled .o files and final ELF
- `config/` - Splat and linker configuration files
- `docs/` - Documentation
- `lib/N64Recomp/` - Git submodule containing the recompiler tool
- `patches/` - Game modifications (future)
- `recompiled/` - Generated C code output (gitignored)
- `src/game/` - Hand-written C code
- `symbols/` - Symbol definitions and stub lists
- `tools/` - Build scripts and tooling

### Hardware Stubbing Strategy

Functions containing these instructions are automatically stubbed:
- COP0 registers: `mfc0`, `mtc0`, `dmfc0`, `dmtc0`, `cfc0`, `ctc0`
- Cache/TLB: `cache`, `tlbp`, `tlbr`, `tlbwi`, `tlbwr`
- System: `eret`, `sync`, `syscall`, `break`
